01 / WHERE YOU START

Module 0 isn't "hello world."

It's a direct translation of what you already know in C into C# syntax — so by lesson six, you're building a linked list you've built before, just without malloc/free to manage yourself.

lessons/00-06-classes-objects/README.md the same linked-list node, two languages
C — WHAT YOU KNOW
struct Node {
    int data;
    struct Node *next;
};

struct Node *make_node(int value) {
    struct Node *n = malloc(sizeof(struct Node));
    n->data = value;
    n->next = NULL;
    return n;
}
// ...and you MUST remember to free(n) for every node.
C# — SAME IDEA
class Node
{
    public int Data;
    public Node? Next; // '?' — same idea as NULL

    public Node(int data) // a constructor
    {
        Data = data;
        Next = null;
    }
}

Node n = new Node(5); // no malloc, no free
Same struct, same allocate-and-link pattern you already know — no sizeof, no manual NULL setup, and crucially, no free(). The .NET garbage collector reclaims the node once nothing references it — you cannot double-free, and you cannot create a dangling pointer to freed memory. This is Lesson 00.6; by 00.7 you're rebuilding a stack on top of List<T>.
02 / WHAT THIS IS

Not "C# for absolute beginners."

This course assumes you already write code — specifically C, and that you already know your way around a linked list and a hash map. Module 0 spends its time translating what you already know (pointers, structs, manual memory management, dynamic arrays) directly into C#'s equivalents, so you're writing real C# by lesson two, not still reading about semicolons.

Every lesson pairs a deep explanation with a working exercise and a full correction. The six projects go further: each ships as a brief with a secure-coding checklist, a starter skeleton, a reference solution, and an xUnit self-check suite — tests that fail until your implementation is actually right, not just until it compiles.

Security isn't a chapter at the end. It's a dedicated module (PBKDF2 password hashing, AES-GCM authenticated encryption, timing-safe comparison, parameterized queries) and a checklist baked into every single project — because SELECT * with a concatenated string is the kind of thing that gets people fired, and you should know that before you ship your first tool, not after.

04 / WHAT'S DIFFERENT

Every lesson shows you both sides.

Most tutorials teach you the pattern that works. This one also shows you the pattern that gets you paged at 3am — and explains, line by line, why one is fine and the other isn't. This exact example is Lesson 22a.

lessons/22a-secure-coding-fundamentals/README.md SQL injection — before / after
DON'T
// the classic vulnerability
string query = $"SELECT * FROM Users WHERE Username = '{username}'";
// if username is:  ' OR '1'='1
// the query becomes:
// SELECT * FROM Users WHERE Username = '' OR '1'='1'
// — returns everything
DO
// parameterized — the DB treats the value as DATA, never SQL
using var command = new SqlCommand(
    "SELECT * FROM Users WHERE Username = @username", connection);
command.Parameters.AddWithValue("@username", username);
Left: string interpolation. A username of ' OR '1'='1 just returned every row in the table. Right: the parameter value is sent separately from the query text — the database cannot confuse data for code. This is the whole lesson, and it's the same principle Project 3's VaultCrypto and Project 6's AesGcmEncryptionService apply to encryption instead of queries.
